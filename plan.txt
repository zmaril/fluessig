fluessig — plan

The canonical design is DESIGN.md (rationale, decisions §11, non-goals §8, derived artifacts §9).
This is the actionable checklist, tracking DESIGN §7's build order.

V1 SCOPE — **COMPLETE: the rip-out list is done.** Schema templates, migrations.rs, parse_pk,
BOOL_COLUMNS, gen_models.py, gen-tables.ts, and ALL THREE hand-written binding lib.rs files
(node/python/ruby) are deleted/generated; every artifact drift-guarded; every suite green.
Original statement: **bindgen + models, replacing entl's hand-written surface.** entl's schema
templates, generated read models (tables.gen.ts / models.py), and hand-written binding glue
(entl-node/entl-python/entl-ruby lib.rs) are the dogfood target and get ripped out in favor of
fluessig output. So the spine is TWO verticals off one entl.tsp:
  models:  catalog.json -> SQL DDL + SQL data codec -> live store   (Steps 1-5)
  bindgen: api.json -> generated napi/PyO3/Magnus glue -> entl's test suites green (Step 5b)
Fan-out (Mongo/ORM-codegen/graph/files) stays post-v1.

Shape: authored .tsp (+ @fluessig/typespec decorators) --tsp compile--> catalog.json (the contract)
--> Rust core (loader/validator + codecs) reads catalog.json (serde), no Node at data time.
Arrow is an import + data path. Mechanism proven in spike/ — ONE entl.tsp holds both layers; one
compile emits catalog.json (model layer) AND api.json (op layer -> generated language bindings,
cargo-check-proven; see spike/README.md + /translation.md).

v1 authoring toolchain: `cargo install --git` for the binary + ambient Node for .tsp->catalog.json
(fine for the entl dogfood). Bundled no-Node authoring binary = a later packaging milestone (§11.4).

## Step 3 — SQL DDL back-end (Postgres first)

[x] sql.rs: tables(&Catalog, Dialect) -> structured TableDefs (the physical projection: relations ->
    FK columns / association+edge tables, families -> per-leaf tables, keys -> PKs incl. FK-in-PK
    expansion and column-sharing dedup) + render() -> CREATE TABLE text in the template style +
    ddl(). PG type map keyed by TypeSpec builtin NAME (the numeric builtins all root at `numeric`,
    so the base is useless for width) with semantic overrides (Oid/Json/bytes -> hex text).
[x] PARITY GATE GREEN — ALL THREE DIALECTS: tests/sql_parity.rs parses every hand-written template
    (postgres 20, sqlite 20, duckdb 28 — the complete engine store) and holds the generated tables
    to them — column set (name -> type + NOT NULL) strict, PK ordered-exact, DEFAULTs
    template-lenient, column ORDER deliberately excluded (sinks address by name).
    Caught on day one: the numeric-base type bug (TypeSpec ints all root at `numeric` — map by
    NAME), PK column-sharing dup, AND a real modeling error (GitObject was over-keyed (oid, repo);
    real PKs key on oid alone — fixed in entl.tsp). Dialect maps: PG/SQLite oids+bytes -> hex text,
    SQLite bools -> 0/1 + timestamps -> RFC3339 text, DuckDB -> raw blob + native TIMESTAMP.
    => entl's ENTIRE hand-written schema surface (all three migrations dirs) now regenerates
    from entl.tsp.
    RETIRED at the Step-5 swap: the templates are deleted, so the gate has nothing to parse —
    its proof stands at the moment of deletion (commit the fluessig work + green gate FIRST,
    the rip-out SECOND, so the equivalence proof is in history). Ongoing drift protection =
    the regenerates-identically test + entl's behavior suites (P1/P2/P3 round-trips).
[x] _fluessig_meta + fingerprint + extras (§9.5): sql::fingerprint(catalog, dialect, extras) =
    sha256 content hash; ddl() emits every table + derived views + a one-row _fluessig_meta
    (fingerprint/format/emitter/compiler, generated_at DB-defaulted so DDL text is deterministic)
    + extras appended verbatim LAST and hashed into the fingerprint (editing extras trips drift).
    Rebuild flow: compare fingerprints; mismatch => drop + re-apply.
[x] Minimal @derived (§9.3 carve-out): @derived("exists"|"count", of, filter?) end-to-end —
    decorator -> emitter -> schema -> IR (Derived) -> validator (agg whitelist; of = a to-many
    relation on the same entity; filter keys = edge properties) -> sql::derived_views(): a
    <table>_derived view per entity (EXISTS/count subquery over the edge table; CREATE VIEW IF
    NOT EXISTS on sqlite). entl.tsp: isMerge carries the real spec; the stored column REMAINS for
    template parity until the Step-5 swap (then: column drops, view is the projection).
    NB: `of` is a string, not a checked reference — a decorator arg can't self-reference its own
    model mid-declaration; the Rust validator resolves it instead (noted vs FINDINGS #2 taste).

## Step 4 — SQL data codec (Layer C; port the driver sink)

[x] data.rs: Mutation{table, op: Insert|Upsert|Delete, columns, rows} in a Transaction ->
    SqlCodec::plan() -> a Plan of Steps. v1 posture: SQL = ONE transactional step; statements
    topologically ordered (parents-before-children for writes, reversed for deletes; association/
    edge tables after both endpoints, abstract endpoints after every leaf; cycles capped).
    Strict checking: unknown table/column and delete-without-full-key are errors, never drops.
    Rows are JSON values in v1 (what entl's Arrow->JSON conversion already produces); the
    zero-copy RecordBatch path lands with Step 2 so the arrow version is aligned deliberately.
[x] upsert_sql (ON CONFLICT (pk) DO UPDATE SET / DO NOTHING; $n for pg, ? for sqlite/duckdb),
    insert_sql, delete_sql (WHERE pk-only, params bound in PK order).
[x] PORT + DELETE parse_pk — as COMMITTED GENERATED CODE, not runtime rendering (decided):
    `fluessig-gen <catalog.json> <out.rs>` (sql::rust_schema_module) emits a self-contained Rust
    module — per-dialect TableSchema consts (name, pk, __table__-templated DDL, all 28 tables x 3
    dialects) — committed as crates/entl-core/src/schema_gen.rs (the tables.gen.ts/models.py
    convention on the Rust side). DriverSink reads it directly; entl-core has NO fluessig runtime
    dependency and does no parsing/rendering at runtime. A regenerates-identically test in
    fluessig guards drift (stale schema_gen.rs fails cargo test with the regen command in the
    message). driver_tables() is catalog-driven (28 tables, was PG_TABLES' 20). Dogfood green:
    entl-core 9/9 + fluessig 21 + napi rebuild + bun test (PGlite end-to-end + coverage).
[ ] ON DELETE CASCADE on composition child tables — moot until a @compose relation exists in a
    catalog (entl's faithful schema has none; FINDINGS #5). Revisit with the Mongo codec.
[ ] Lenient coercion mode (documented whitelist: numeric widening, absent-nullable -> null).

## Testing (cross-cutting, not a phase) — DESIGN §7 / §11.13

    artifact) and then fails if `git status --porcelain` is non-empty — uncommitted codegen drift

    Node-side emit step that cargo test can't run.)
